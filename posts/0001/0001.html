Introducing My New Blog

The first blog that I wrote in was in blogspot, or blogger (I'm still not sure what the difference is between them). I then tried Wordpress, before finally settling on blogger again. To be fair, the new blogger is pretty good, and I do like the minimalistic theme. However, I wanted absolute, complete control over my website. Why? Ownership breeds pride. I won't put in personal manual labor improving the garden of my rented house. But if it was my own, I most certainly would.

Similarly, with complete ownership over my content and how it's displayed, I can put in effort into designing it better. I mean CSS-styles better. I mean having the option to use superb UI frameworks like Foundation.

I've decided to start with a very simplistic, very very minimalistic design though. I will be playing around with CSS to make it look nicer, but I won't be putting in any flashy stuff. My inspiration is the indomitable Mike Bostock, whose blog made me realize that features does not equal beauty. (He also resolved the very important problem of naming one's blog: simply use your full name).

In the interest of complete disclosure, my previous blog, which I also tried to be as simplistic as possible, is here. Since there appears no way to de-host it, I'm apparently stuck with the horrifying memories of my rookie web design skills for ever.

Anyways, yes, I did get "inspiration" (read: source code) from Mike's blog. He doesn't allow copying the code verbatim, so I only looked at how he was working with the CSS. The hope is to make this blog simplistic and not a clone.

Interestingly, that's another reason why I've chosen to go commando, blog wise. Webpages are representations of our digital selves, and hence should reflect the values that we hold near and dear. Hence, the attempt at simplicity and cleanliness.

I will most likely be adding improvements constantly. I hope to not use any external frameworks, except for adding functionality such as code display, image lightboxes and such. One of the things I really really want to do is learn D3 and make cool visualizations and show them to the world right here.

The process for creating a page is to create an HTML file manually. It is then pushed to Github, along with changes to the main page to link to that page. Its abysmally manual, but I promise to add simple automation later. I just really wanted to get something out of the house. Also, hopefully the minimal design will be easier to read for anyone trying to implement something similar.

Wish me luck!

Configuring A Custom Domain

(Added August 8 2014)

One of the first things I wanted was a custom domain name. There are many reasons why you would want to host your website under a custom domain name rather than using Github's subdomain, but I think the most important one is that in case I wanted to move my site away from Github, I would have to change all references to my old webpage: Resume's, other websites, business cards etc. Having you own domain name can be really cheap, I used Namecheap to obtain mine for $8 p.a.

I wasn't really familiar with DNS, so at first I had trouble configuring it. In particular, I did not configure an A record to point to the ip address of Github. An A record is an address record, which is what tells your DNS host (in this case, Namecheap) where he can find the content for the domain. Since I did not configure it, of course the content could not be found.

Anyways, here's what you need to do:

  1. add a CNAME file in your repository which contains the bare domain name
  2. configure two A records to point to the ip address of Github pages in Namecheap. You can find the ip addresses here.
  3. Add a CNAME for the url provided by Github in Namecheap, so that everyone that tries to access that url will be redirected to your new page. So, my case I added a CNAME record with url as pratikmallya.github.io. Therefore, anyone entering that url in the browser will be automatically redirected to my custom domain name pratikmallya.me.
